1. **Security data leak** — credentials/keys/PII committed to the repo, including connection strings in DAG configs, dbt `profiles.yml` checked in by mistake, service-account JSONs in repo, Airflow Variables echoed to logs, or PII surfacing in fixture/seed data.
2. **Security bugs** — SQL/template injection in dynamic queries, `f"..."` interpolation of user-supplied IDs into SQL, SSRF in extract steps that hit user-controlled URLs, unsafe `pickle.load` on artefact files, path traversal in S3/GCS key construction, broken TLS verification.
3. **Correctness bugs** — silent type coercion in pandas/Spark (`object` columns hiding `None`s, `int64` → `float64` from NaN), wrong join key/granularity producing row blow-ups, off-by-one window boundaries, incorrect timezone handling (mixing naive + aware datetimes, UTC vs local), nullability assumptions that break when a column is empty, swallowed exceptions in retry loops.
4. **Data-loss / blast-radius risks** — destructive SQL without a transaction or backfill plan (`DROP`/`TRUNCATE`/`DELETE` without `WHERE`), `MERGE` statements that overwrite history, idempotency missing on writes (re-runs duplicate or replace rows), schema changes that nullify existing rows, dropped columns still referenced downstream, unbounded fan-out in `expand_kwargs` / dynamic task mapping that can swamp the executor.
5. **Pipeline correctness** — DAG dependencies wrong (downstream reads from upstream before completion), incorrect `schedule_interval` / `catchup` defaults producing duplicate or missing runs, retries on non-idempotent operators, watermark / late-arrival handling missing on event streams, partition pruning broken by predicate placement.
6. **Performance footguns** — N+1 queries in row-by-row loops where a single set-based query would work, missing indexes on new lookup columns, full-table scans on partitioned tables (predicate not on partition key), `collect()` / `toPandas()` on Spark DataFrames in production paths, Airflow operators doing heavy compute in the scheduler process.
